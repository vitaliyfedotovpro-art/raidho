## Топ-3 рекомендации по улучшению качества кода

### 1. Исправить «проглатывание» исключений и ошибочный `except` в горячих путях

В `loop.py` в методе `code()` стоит:

```python
except (ProcedureError, Exception):
    pass
```

`except (ProcedureError, Exception)` эквивалентно `except Exception` (т.к. `ProcedureError` — наследник `Exception`), то есть перечисление вводит в заблуждение. Хуже того, голый `pass` без логирования молча скрывает любой сбой процедуры, включая баги в коде, а не только ожидаемые `ProcedureError`. Аналогичные «немые» `except Exception` есть в `memory.recall()` (возврат `""` при любой ошибке поиска) и в `providers.py` (`json.loads(...) → fargs = {}`).

**Что сделать:** ловить узкий тип (`except ProcedureError`), а непредвиденные исключения как минимум логировать (`logging.exception(...)`) перед фолбэком. Это сразу повысит отлаживаемость без изменения поведения для пользователя.

### 2. Закрыть docstring у публичного и «контрактного» API (а не у всех 36 функций подряд)

Не нужно слепо добивать покрытие до 100% — нужно точечно задокументировать функции, образующие внешний контракт:
- `AgentMemory.remember`, `AgentMemory.n_facts` — публичный API класса памяти;
- переопределения `chat`/`agent_turn` в `AnthropicProvider` и `OpenAICompatProvider` — у них есть нетривиальные различия (Anthropic использует `thinking`/`tool_use`, OpenAI-совместимый — `tool_calls` и ручной разбор JSON), которые стоит зафиксировать в коде;
- `repl` и `main` в `cli.py` — точки входа.

Внутренние однострочники (`_render`, `_resolve`, `_print_tool`, вложенные `_llm`/`_bash`/`_exec`) документировать не обязательно. Такой подход уберёт ~15–20 из 36 «дыр», сохранив сигнал и не размывая код шумовыми комментариями.

### 3. Устранить риски безопасности/устойчивости в I/O и сетевом слое

- **`tools.py` / `bash`:** команды выполняются `create_subprocess_shell` без таймаута — зависшая команда (`npm install`, интерактивный prompt) блокирует сессию навсегда. Стоит добавить `asyncio.wait_for(...)` с разумным лимитом и обработкой `TimeoutError`. Это прямо заявлено как «unsandboxed, full access», поэтому хотя бы защита от зависаний обязательна.
- **`providers.py` / `_post`:** ответ `httpx` парсится как `r.json()` без проверки HTTP-статуса; при 4xx/5xx с не-JSON телом будет невнятный креш вместо аккуратного `[LLM error: ...]`. Добавить проверку `r.status_code` / `try/except` вокруг `r.json()`.
- **Мелочь-несогласованность:** функция `_preview` в `providers.py` определена, но фактически дублирует `_print_tool` из `loop.py` и, судя по коду, нигде не используется — её стоит либо задействовать, либо удалить (мёртвый код).
